## Summary

Phase 10 is a pure data-layer and infrastructure phase with no user-facing endpoints. It has three independent work streams: (1) adding a `source` field to the existing `CriterionScore` model and updating `compute_overall_band` to filter on it, (2) creating the new `AIFeedbackJob` model to track async job lifecycle, and (3) installing and wiring up django-q2 with the ORM broker.

All three streams follow patterns already established in the codebase. The `source` field should be an `IntegerChoices` enum (same pattern as `SessionStatus`, `SpeakingCriterion`). The `AIFeedbackJob` model uses `TimestampedModel` as base (same as all other session models). django-q2 requires only settings + `INSTALLED_APPS` change plus `python manage.py migrate` — no new infrastructure.

The most critical correctness concern is the `unique_together` constraint on `CriterionScore`. Currently it is `("session_result", "criterion")`. After adding `source`, the same `session_result` + `criterion` combination can legitimately exist twice (once for EXAMINER, once for AI). The constraint must be updated to `("session_result", "criterion", "source")`. The migration must handle this by first removing the old constraint and adding the new one in a single migration — this is a safe operation because no AI rows exist yet.

**Primary recommendation:** Three migrations total — one for CriterionScore (add source, update unique_together), one auto-generated by django-q2 install (run migrate), one for AIFeedbackJob. Task skeleton goes in `session/tasks.py`.

### Pattern 1: ScoreSource IntegerChoices Enum
**What:** Add `ScoreSource` as an `IntegerChoices` class in `session/models.py`, parallel to existing `SessionStatus` and `SpeakingCriterion`.
**When to use:** Whenever a model field needs human-readable choices with integer storage.
**Example:**
```python
# session/models.py — add near other choices classes (line ~24)
class ScoreSource(models.IntegerChoices):
    EXAMINER = 1, "Examiner"
    AI = 2, "AI"
```

### Pattern 2: CriterionScore source field + updated unique_together
**What:** Add source field with default EXAMINER (so existing rows are valid without a data migration). Expand unique_together constraint.
**When to use:** The migration must drop the old constraint and add the new one atomically.
**Example:**
```python
class CriterionScore(TimestampedModel):
    session_result = models.ForeignKey(SessionResult, on_delete=models.CASCADE, related_name="scores")
    criterion = models.PositiveSmallIntegerField(choices=SpeakingCriterion.choices)
    source = models.PositiveSmallIntegerField(
        choices=ScoreSource.choices,
        default=ScoreSource.EXAMINER,
    )
    band = models.PositiveSmallIntegerField(validators=[MinValueValidator(1), MaxValueValidator(9)])
    feedback = models.TextField(null=True, blank=True)

    class Meta:
        unique_together = [("session_result", "criterion", "source")]
```

### Pattern 3: compute_overall_band EXAMINER-only filter
**What:** Filter the queryset to EXAMINER source before computing the band average. No change to the arithmetic.
**When to use:** Required to prevent AI scores from polluting the examiner-computed overall band.
**Example:**
```python
def compute_overall_band(self):
    bands = list(
        self.scores.filter(source=ScoreSource.EXAMINER).values_list("band", flat=True)
    )
    if len(bands) < 4:
        return None
    return (sum(bands) // 2) / 2
```

### Pattern 4: AIFeedbackJob model
**What:** New model tracking async job lifecycle, FK to `IELTSMockSession`.
**When to use:** One `AIFeedbackJob` row per AI feedback request. Status field uses `IntegerChoices`.
**Example:**
```python
class AIFeedbackJob(TimestampedModel):
    class Status(models.IntegerChoices):
        PENDING = 1, "Pending"
        PROCESSING = 2, "Processing"
        DONE = 3, "Done"
        FAILED = 4, "Failed"

    session = models.ForeignKey(
        IELTSMockSession,
        on_delete=models.CASCADE,
        related_name="ai_feedback_jobs",
    )
    status = models.PositiveSmallIntegerField(
        choices=Status.choices,
        default=Status.PENDING,
        db_index=True,
    )
    error_message = models.TextField(null=True, blank=True)

    def __str__(self):
        return f"AIFeedbackJob({self.pk}) session={self.session_id} status={self.get_status_display()}"
```

### Pattern 6: Task skeleton in session/tasks.py
**What:** A plain importable Python function that django-q2 can enqueue by dotted path or direct reference.
**When to use:** The skeleton must be `async_task`-safe — no Django request context, all IDs passed as arguments.
**Example:**
```python
# session/tasks.py
import logging

logger = logging.getLogger(__name__)


def run_ai_feedback(job_id: int) -> None:
    """
    Background task: transcribe audio and generate AI feedback for a session.
    Enqueued via: async_task('session.tasks.run_ai_feedback', job_id)
    """
    from session.models import AIFeedbackJob

    try:
        job = AIFeedbackJob.objects.get(pk=job_id)
        job.status = AIFeedbackJob.Status.PROCESSING
        job.save(update_fields=["status", "updated_at"])

        # Phase 11: transcription
        # Phase 12: AI scoring
        logger.info("run_ai_feedback job_id=%s — skeleton placeholder", job_id)

        job.status = AIFeedbackJob.Status.DONE
        job.save(update_fields=["status", "updated_at"])

    except AIFeedbackJob.DoesNotExist:
        logger.error("run_ai_feedback: job_id=%s not found", job_id)
    except Exception as exc:
        logger.exception("run_ai_feedback job_id=%s failed: %s", job_id, exc)
        try:
            job.status = AIFeedbackJob.Status.FAILED
            job.error_message = str(exc)
            job.save(update_fields=["status", "error_message", "updated_at"])
        except Exception:
            pass
```

### Anti-Patterns to Avoid
- **Calling `async_task` inside a transaction.atomic block:** django-q2 enqueues by writing to the ORM — if the outer transaction rolls back, the task row is also rolled back, but the in-memory state may have advanced. Follow the existing project discipline: call `async_task` after the transaction exits (same rule as `_broadcast`).
- **Changing compute_overall_band without a filter:** Without the `filter(source=ScoreSource.EXAMINER)`, AI scores submitted in Phase 12 would corrupt the examiner's overall band. This is AIAS-05.
- **Setting default=None on source field:** The field needs `default=ScoreSource.EXAMINER` so existing rows survive the migration without a data migration step.
- **Workers > CPU cores for CPU-bound tasks:** In Phase 11, Whisper will be CPU-heavy; keep `workers=2` for this phase's skeleton to avoid over-provisioning.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Task queue persistence | Custom DB table + polling loop | django-q2 ORM broker | Race conditions, retry logic, dead-letter handling are already solved |
| Task status tracking | Custom status table | `AIFeedbackJob.status` + django-q2's own task table | django-q2 has a `Task` model in admin; `AIFeedbackJob` provides app-level status |
| Worker process management | Custom subprocess launcher | `python manage.py qcluster` | django-q2 manages worker pool, signals, and graceful shutdown |

**Key insight:** The ORM broker in django-q2 is specifically designed for projects that can't or won't run Redis. It uses a polling model against the Django database and requires zero additional infrastructure beyond `pip install django-q2` + migrate.

## Common Pitfalls

### Pitfall 1: unique_together not updated in migration
**What goes wrong:** Adding `source` but leaving `unique_together = [("session_result", "criterion")]` means two AI scores for the same criterion on the same result will cause a database IntegrityError, not a clean ValidationError.
**Why it happens:** Django does not automatically update unique_together when you add a new field.
**How to avoid:** The migration must alter_unique_together — remove `("session_result", "criterion")` and add `("session_result", "criterion", "source")` in the same migration file.
**Warning signs:** `IntegrityError` when trying to insert a second CriterionScore for the same result+criterion with a different source.

### Pitfall 2: update_or_create in views.py no longer covers AI scores
**What goes wrong:** The existing `CriterionScore.objects.update_or_create(session_result=result, criterion=score_data["criterion"], ...)` in `SubmitResultView` does not specify `source`. After the unique_together change to include source, the lookup defaults to whatever `source` the model default is — which is EXAMINER. This is correct for the examiner submit endpoint and requires no change.
**Why it happens:** The `update_or_create` defaults will hit `source=EXAMINER` implicitly (Django model default). The AI submission path (Phase 14) will need to explicitly pass `source=ScoreSource.AI`.
**How to avoid:** In Phase 10, no change to views.py is needed. Document this for Phase 14 planning.
**Warning signs:** Phase 14 AI scores accidentally overwrite examiner scores if `source` is not passed explicitly.

### Pitfall 3: django-q2 qcluster fails because django_q is not in INSTALLED_APPS
**What goes wrong:** `python manage.py qcluster` raises `CommandError` or `django_q` migrations are missing.
**Why it happens:** `django_q` must be in `INSTALLED_APPS` AND `python manage.py migrate` must be run before qcluster works.
**How to avoid:** Add `django_q` to `INSTALLED_APPS`, run migrate, then test `python manage.py qcluster`.
**Warning signs:** `No such table: django_q_task` error.

### Pitfall 4: async_task called before transaction commits
**What goes wrong:** The task is enqueued (ORM row written), but the outer transaction that created `AIFeedbackJob` rolls back. The worker then tries to fetch a non-existent job.
**Why it happens:** ORM broker enqueues by inserting a row — it participates in the current transaction.
**How to avoid:** Call `async_task(...)` after `transaction.atomic()` block exits. This is already established project discipline (same as `_broadcast`).
**Warning signs:** Worker logs "job_id=X not found" shortly after an enqueue.

### Pitfall 5: sync mode in tests
**What goes wrong:** Tests that call `async_task` spin up real workers and hit external services.
**Why it happens:** django-q2 runs tasks synchronously when `Q_CLUSTER = {'sync': True}` — but this must be set in `settings_test.py`.
**How to avoid:** Add `Q_CLUSTER = {'sync': True}` to `MockIT/settings_test.py`. Tests using `async_task` will then execute the function immediately in-process.
**Warning signs:** Tests hang waiting for qcluster, or tests fail with "channel layer" or "broker" errors.
